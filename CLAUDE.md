# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Human-Agent Teamwork Search and Rescue simulation for TU Delft Collaborative AI. Uses MATRX v2.2.0 for a 2D grid environment where RescueBot(s) and a human agent cooperate to rescue victims.

**MATRX directories**: `matrx/` is the locally modified copy (`grid_world.py`, `defaults.py` are modified here — safe to touch). `matrx-master/` is the unmodified upstream source — do not modify it.

Active agent architecture:
- **SearchRescueAgent** (`agents1/search_rescue_agent.py`): Primary LLM agent using Ollama (SDK or REST), OpenAI-style tool calling, shared memory, modular perception
- **Baseline Agent** (`agents1/OfficialAgent.py`): Rule-based FSM with trust belief system (being phased out)

The old `RescueAgent.py` and its `PerceptionModule.py` have been archived to `agents1/agents_graveyard/`.

## Running the Simulation

```bash
# Requires Ollama running locally with qwen3:8b pulled
ollama pull qwen3:8b

pip install -r requirements.txt
# Also install unlisted runtime deps:
pip install requests pyyaml

python main.py
# Then open http://localhost:3000, Ctrl+F5 to clear cache
# Use 'God' view to start; human agent view for keyboard control
```

**No test suite exists.** Testing is done by running `main.py` and observing simulation output and `logs/score.json`.

**Configuration** is hardcoded in `main.py`:
```python
condition = "normal"                    # "normal" | "strong" | "weak"
name = "humanagent"                     # human agent name
agent_type = 'marble'                   # 'marble' | 'llm' | 'langgraph' | 'baseline'
ticks_per_iteration = 1200              # ticks before EnginePlanner generates new tasks
num_rescue_agents = 2                   # number of AI agent instances (1–5)
include_human = False                   # True to add keyboard-controlled human agent
ollama_base_port = 11434                # Ollama port (single instance for marble)
planner_model = 'qwen3:8b'             # model for EnginePlanner
manual_plans_file = "manual_plans.yaml" # set None to use LLM task generation
planning_mode = 'dag'                   # 'simple' (flat list) | 'dag' (task graph with conditionals)
agent_presets = ['scout', 'medic']      # capability presets per agent (cycles); or custom dicts
capability_knowledge = 'informed'       # 'informed' (know upfront) | 'discovery' (learn from failure)
comm_strategies = ['always_respond', 'always_respond']  # 'always_respond' | 'busy_aware'
llm_backend = 'ollama_sdk'              # 'ollama_sdk' (Ollama Python SDK) | 'requests' (direct HTTP)
```

For `agent_type='marble'` (default), **a single Ollama instance handles all agents** — no need for multiple Ollama processes. The old multi-Ollama setup in `HOW_TO_RUN.md` only applies to `agent_type='llm'`. LLM backend is configurable: `'ollama_sdk'` (default, uses Ollama Python SDK) or `'requests'` (direct HTTP to Ollama's OpenAI-compatible endpoint). Set via `llm_backend` in `main.py`.

## Architecture

### Simulation Loop

`main.py` → `create_builder()` → `builder.startup()` → `world.run_with_planner(planner, agents, ticks_per_iteration)`

`run_with_planner()` is a custom method in MATRX's `GridWorld` (`grid_world.py:253`). Each iteration:
1. `EnginePlanner.submit_generate_tasks()` → async LLM call assigns tasks per agent (or reads `manual_plans.yaml`)
2. `SearchRescueAgent.set_current_task(task)` → resets nav/LLM state, preserves memory
3. Run `ticks_per_iteration` MATRX ticks (each tick calls `decide_on_actions()`)
4. `EnginePlanner.submit_summarize()` → async LLM summarizes iteration
5. `EnginePlanner.decide_next_step()` → terminates if `block_hit_rate >= 1.0` or max iterations reached

### Agent Tick Loop (`decide_on_actions` in `LLMAgentBase`)

Called every tick. Phases executed sequentially:

1. **`update_knowledge()`**: Updates SharedMemory with new victim/obstacle discoveries.
2. **`_run_preamble()`**: Navigation — if `_nav_target` set, `Navigator.get_move_action()` returns next step.
3. **`check_if_llm_response_ready()`**: Non-blocking poll `get_llm_result(future)`. If done, dispatch via `execute_action(name, args)`. Plain-text fallback: `ActionMapper._extract_json()` → `execute_action()`.
4. **`_apply_navigation()`**: If LLM returned `MoveTo`/`MoveToArea`/`NavigateToDropZone`, set `_nav_target`.
5. **`_apply_communication()`**: Sends outbound messages generated by `CommunicationModule`.
6. **Submit LLM call**: If `_reasoning_step=True` and no pending future, `build_prompt()` → `submit_llm_call()`. Returns `Idle` while waiting.

**Perception**: `filter_observations()` limits state to Chebyshev distance ≤ 1 (1-block radius), always including self, `'World'`, `'humanagent'`, doors.

**`_reasoning_step` flag**: Set to `True` to trigger a new LLM call next tick. This is the gate between "need LLM" and "LLM pending" states.

### Key Modules (`agents1/`)

- **`llm_agent_base.py`**: Unified infrastructure base class for all LLM agents. Inherits `ArtificialBrain` + `Perception` mixin. Handles A* navigation, cooperative carry retry (timeout: `CARRY_WAIT_TIMEOUT_TICKS = 100`), SharedMemory rendezvous, LLM polling lifecycle, communication dispatch, and task injection. Contains `AREAS_CONFIG` (14 named rescue areas).

- **`search_rescue_agent.py`**: Primary LLM agent. Inherits `LLMAgentBase`. Configures Ollama model, reasoning strategy, capability presets, and tool schemas. Instantiates `ReasoningIO` and `CriticBase`.

- **`async_model_prompting.py`**: Non-blocking LLM wrapper via `ThreadPoolExecutor`. Supports two backends: Ollama Python SDK (`ollama_sdk`) and direct HTTP (`requests`). `submit_llm_call()` returns a `Future` immediately; `get_llm_result()` polls without blocking. Configure via `set_backend()` or `LLM_BACKEND` env var.

- **`execution_module.py`**: Unified action dispatcher — maps `(action_name, args)` → MATRX `(action_class_name, kwargs)`. Automatically enriches cooperative actions with `partner_name`. 16 supported actions: directional moves, `MoveTo`, `MoveToArea`, `EnterArea`, `NavigateToDropZone`, `SendMessage`, `CarryObject`, `Drop`, `CarryObjectTogether`, `DropObjectTogether`, `RemoveObject`, `RemoveObjectTogether`, `Idle`.

- **`action_mapper.py`**: Extracts JSON from free-form LLM text with three fallback strategies: fenced ` ```json` blocks → first `{...}` span → `ast.literal_eval` dict literal.

- **`prompt_builder.py`**: Constructs OpenAI-style message lists from reasoning strategy, task, perceived state, and shared memory context. Supports `'cot'`, `'react'`, `'reflexion'` strategies.

- **`tool_registry.py`**: 13 LangChain `@tool`-decorated action functions. Builds OpenAI-compatible tool schemas via `convert_to_openai_tool`. Also contains `REASONING_STRATEGIES` dict and `GAME_RULES` constant.

- **`modules/perception_module.py`** (lowercase): `Perception` mixin. `percept_state()` converts filtered MATRX state to compact JSON for LLM prompts. Classifies 7 object types: victim, rock, stone, tree, door, agent, wall.

- **`modules/communication_module.py`**: Environment-sided message processor. Pluggable strategies: `AlwaysRespondStrategy`, `BusyAwareStrategy`. Async LLM summarization when message count exceeds threshold (default 10). Methods: `process_messages()`, `get_messages_for_prompt()`, `has_pending_ask_help()`. Note: `modules/__init__.py` still exports `CommunicationModule` from `agents_graveyard` — the active module is this lowercase file.

- **`modules/planning_module.py`**: Hierarchical task decomposer. Two modes: `'simple'` (flat `SubTask` list) and `'dag'` (task graph with conditional branching). `TaskGraph` parses conditional syntax; `TaskNode` + `TaskStatus` enum (PENDING/ACTIVE/COMPLETED/SKIPPED). `plan()` / `poll_plan()` / `replan()` lifecycle.

- **`capabilities.py`**: Agent capability differentiation. Four dimensions: Vision (1–3 blocks), Strength (low/medium/high), Medical (low/high), Speed (slow/normal/fast). Presets: `'scout'`, `'medic'`, `'heavy_lifter'`, `'generalist'`. Key functions: `resolve_capabilities()`, `get_capability_prompt()` (LLM descriptions), `filter_tools_for_capabilities()` (restricts tool schemas).

- **`modules/task_critic_module.py`**: `CriticBase` evaluates if the agent has met task requirements. Returns `{"reasoning", "success": bool, "critique"}`. Instantiated in `SearchRescueAgent` but not yet wired into the tick loop.

- **`modules/ReasoningModule.py`**: Older LLM reasoning module (less used in current SearchRescueAgent flow). `ReasoningIO` wraps `query_llm_with_tools_async`.

### Supporting Directories

- **`brains1/`**: MATRX agent controllers. `ArtificialBrain.py` — base class; `decide_on_action()` (singular) wraps `decide_on_actions()` (plural), adding grab_range/max_objects/durations. **Always override the plural form.** `HumanBrain.py` — routes keyboard input.
- **`actions1/CustomActions.py`**: `CarryObjectTogether`, `DropObjectTogether`, `RemoveObjectTogether` — require `human_name` kwarg.
- **`loggers/`**: `ActionLogger.py` — per-tick logger (auto-called by MATRX). `OutputLogger.py` — final summary at run end.

### LLM Integration

**SearchRescueAgent** (marble): calls go through `agents1/async_model_prompting.py` → Ollama SDK or REST API → `qwen3:8b`. Single Ollama instance serves all agents.

**EnginePlanner + memory compression**: calls go through `engine/llm_utils.py` → Ollama REST API directly.

```python
# llm_utils.py async (returns Future):
future = query_llm_async(model, system_prompt, user_prompt, max_tokens, temperature)
# llm_utils.py sync (blocks):
result = query_llm(model, system_prompt, user_prompt, max_tokens, temperature)
# Scale thread pool (call once in main.py):
init_llm_pool(num_agents)  # max(4, num_agents * 3) workers
```

`engine/toon_utils.py` provides TOON serialization (~30–60% token reduction vs JSON) — used in `EnginePlanner._summarize_sync()`.

**Few-shot examples** in `few_shot_examples.yaml` are injected via `load_few_shot(key)`. Categories: `reasoning`, `generate_tasks`, `summarize`, `planning_decompose`, `planning_evaluate`, `communication`, `comm_inbound`, `memory_extract`.

**Prompts**:
- `agents1/prompts_rescue_agent.yaml`: agent reasoning, planning, memory, communication prompts
- `engine/prompts_engine_planner.yaml`: `generate_tasks_*`, `summarize_*`, `default_task_description`

### EnginePlanner (`engine/engine_planner.py`)

Coordinator with prefetch: next iteration's tasks are generated in the background while current ticks run. Supports `manual_plans.yaml` override to skip LLM task generation.

`decide_next_step()` is rule-based: terminates on `block_hit_rate >= 1.0` or `iteration >= max_iterations`.

Logs written to: `logs/exp_<condition>_at_time_<HH>h-<MM>m-<SS>s_date_<DD>d-<MM>m-<YY>y/`. Each run creates `score.json` (updated every tick) and `iteration_history.json` (saved on exit).

### Memory System (`memory/`)

- **`BaseMemory`** (`base_memory.py`): Simple list store — used by `SearchRescueAgent`.
- **`ShortTermMemory`** (`short_term_memory.py`): Typed-dict entries with auto-compression via LLM when capacity (20) exceeded — used by the archived `RescueAgent`.
- **`SharedMemory`** (`shared_memory.py`): Thread-safe (`threading.Lock`) cross-agent discovery cache — **actively used** in `SearchRescueAgent` runs. One instance per simulation, shared across all agents.

`LongTermMemory` (embedding-based, OpenAI API) exists but is **not used** in any active simulation path.

### Obstacle and Victim Rules

| Object | Who can remove/carry |
|---|---|
| Small stone | Either agent alone |
| Tree | RescueBot only |
| Big grey rock | Both agents together (cooperative) |
| Mildly injured victim | Human alone (normal/strong) or cooperative (weak) |
| Critically injured victim | Both agents together always |

RescueBot can only carry one victim at a time.

### Scoring

`CollectionGoal.__progress` tracks `block_hit_rate` (0.0–1.0). 8 target victims: 4 critical (+6 pts each), 4 mild (+3 pts each). Healthy victims score 0.

### Key Architecture Gotchas

- `create_builder()` returns `(builder, agents)` tuple — `agents` contains **only AI agents**, not the human.
- `decide_on_action` (singular, base class) wraps `decide_on_actions` (plural, subclass). **Always override the plural form.**
- Human agent is routed through `is_human_agent` check in MATRX's `__step()` — never goes through the AI path.
- Cooperative actions need `human_name` kwarg — `execution_module.py` injects this automatically.
- Ollama tool calling may fall back to plain-text JSON — `action_mapper.py` handles all three extraction strategies.

## Assignments

Focus on `agents1/` (and optionally `brains1/`, `worlds1/`). Do not modify `matrx-master/` (safe to modify `matrx/`).

**Trust assignment**: Extend `_trustBelief()` in `OfficialAgent.py` (baseline only). Trust beliefs persist in `beliefs/currentTrustBelief.csv` and `beliefs/allTrustBeliefs.csv`.

**Interdependence**: Three human conditions (`normal`/`strong`/`weak`) vary perception range and carrying capability — configured in `worlds1/WorldBuilder.py`.

| Condition | Perception range | Stone removal | Mild victim carry |
|---|---|---|---|
| strong | 10 cells | Solo (or coop, faster) | All at once, solo |
| normal | 1 cell | Solo (or coop, faster) | Solo, one at a time |
| weak | 1 cell | Requires RescueBot coop | Requires RescueBot coop |
